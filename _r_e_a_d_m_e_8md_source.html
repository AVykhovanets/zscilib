3 # Zephyr Scientific Library (zscilib)
4 
5 [![API Documentation](doc/img/badge_api.svg)][1]
6 
7 [1]: https://zscilib.github.io/zscilib/
8 
9 The **Zephyr Scientific Library (zscilib)** is an attempt to provide a set of
10 functions useful for scientific computing, data analysis and data manipulation
11 in the context of resource constrained embedded hardware devices.
12 
13 It is written entirely in C, and while the main development target for the
14 library is the [Zephyr Project](https://github.com/zephyrproject-rtos/zephyr),
15 it tries to be as portable as possible, and a standalone reference project
16 is included to use this library in non-Zephyr-based projects.
17

81 ### Note: Float Stack Usage in Zephyr
82 
83 The sample code in this library typically has the `CONFIG_FLOAT` option set,
84 meaning that floating-point support is configured for
85 [Unshared FP registers mode][1]. This mode is used when the application
86 has a **single thread** that uses floating point registers.
87 
88 If your application makes use of **multiple threads**, and more than one of
89 these threads uses floating-point operations, you should also enable the
90 `CONFIG_FP_SHARING` config flag, which configures the kernel for
91 [Shared FP registers mode][2]. In this mode, the floating point registers are
92 saved and restored during each context switch, even when the associated threads
93 are not using them. This feature comes at the expense of an extra 72 bytes of
94 stack memory per stack frame (`s0..s15` + `FPCSR`, plus an alignment word
95 to ensure that the stack pointer is double-word aligned).
96 
97 [1]: https://github.com/zephyrproject-rtos/zephyr/blob/master/doc/reference/kernel/other/float.rst#unshared-fp-registers-mode
98 [2]: https://github.com/zephyrproject-rtos/zephyr/blob/master/doc/reference/kernel/other/float.rst#shared-fp-registers-mode

289 ## Motivation
290 
291 As the processing power of small, embedded MCUs increases and costs fall, more
292 computation can be done on the endnode itself. This allows for more of the
293 'complex' data analysis that used to take place at the PC or server level
294 (data aggregation, statistical analysis, etc.) to be done in less time,
295 using less data storage, and at a lower overall processing cost.
296 
297 > A key goal of zscilib is to allow more data processing to happen on the
298  endnode.
299 
300 By generating immediately actionable and scientifically-relevant data points
301 (standard SI units, pre-filtered data, etc.) directly on the endnode, zscilib
302 aims to be a bridge between raw data and more numerically complex toolkits like
303 `gsl`, `numpy` or `R`.
304 
305 #### What makes zscilib distinct?
306 
307 Numerous high quality, mature, open source scientific libraries already exist:
308 
309 - [GNU scientific library (gsl)](https://www.gnu.org/software/gsl/)
310 - [Lis (Library of Iterative Solvers for linear systems)](https://github.com/anishida/lis)
311 - [CMSIS-DSP](http://www.keil.com/pack/doc/CMSIS/DSP/html/index.html)
312 
313 Despite the wealth of mature functions in these existing libraries, though,
314 they tend to have the following two problems in an embedded context:
315 
316 - They are overly broad and resource intensive (GSL, etc.), and thus aren't
317  appropriate for small, resource constrained devices like the ARM Cortex M
318  family.
319 - They are missing many of the domain-specific features required to convert
320  raw sensor data into actionable information (CMSIS-DAP, Lis).
321 
322 The second item is of particular importance, since the goal of embedded systems
323 is often 'sensing' via raw data, correlating that data, and acting on the final
324 data points or passing them on for further analysis.
325 
326 CMSIS-DAP contains a number of highly efficient algorithms for filtering
327 raw sensor data, but it doesn't offer any domain-specific assistance converting
328 filtered accelerometer vectors into orientation data, for example, or reading
329 a set of photodiodes and converting that data into a useful photometric value
330 like lux. It is excellent at 'conditioning' data, but not at 'understanding' it.
331 
332 zscilib aims to find a middle ground between these two, allowing for richer
333 processing of raw data, but within the confines and limitations of the class
334 of microcontrollers commonly used on low-cost sensor endnodes.
335 
336 ## Architecture-Specific Optimisations
337 
338 Basic tooling has been added to allow for optimised architecture-specific
339 implementations of key functions in assembly.
340 
341 At present, this feature isn't being actively used or developed, but an aim
342 of zscilib is to add optimised versions of key functions to try to get the
343 best possible performance out of limited resources.
344 
345 Initial optimisation will target the **ARM Cortex-M** family of devices and the
346 **Thumb** and **Thumb-2** instruction sets, though other architectures can be
347 accommodated if necessary or useful.
